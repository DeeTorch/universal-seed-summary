Overview

What Is USS?

The Universal Seed Summary Invoker (USS) is a formal prompt protocol designed by Jusstin DeRemus (The Oracle). It acts as a meta-synthesizer ‚Äî an instruction layer you inject into any LLM to transform a full conversation thread into a compressed, structured, machine-readable artifact. Think of it as a "save state" for AI conversations.

üß¨ What It Produces

USS generates a YAML-fronted Markdown document containing a locked record of your thread: its architecture decisions, open questions, failure flags, knowledge topology, and a Resurrection Hook ‚Äî a single sentence that can rehydrate any LLM back into the exact context of the prior session.

üîí What It Protects Against

Context loss across sessions, hallucinated continuations when resuming conversations, inconsistent summaries between models, and undocumented architectural drift. USS enforces "truth-over-helpfulness" as a hard constraint ‚Äî no invention, no inference outside thread boundaries.

‚öôÔ∏è What It Is Not

USS is not a chatbot plugin, not a memory database, and not a summarization tool. It is a prompt-level protocol ‚Äî a schema and a set of constraints you invoke inside any capable LLM to produce a canonical audit artifact from a conversation thread.

üåê Where It Lives

USS runs inside the LLM itself, triggered by pasting the protocol into any session. Artifacts output by USS are portable Markdown files that can be committed to GitHub, stored in Obsidian, ingested by other LLMs, or archived in any knowledge management system.

Architecture

The 8-Section Schema

Every USS artifact follows a rigid, ordered schema. Each section is either required or conditionally required. The schema is identical across all four model wrappers ‚Äî only the delivery scaffolding changes per model.

1

HEADER ‚Äî Thread Lock & Audit

Establishes the thread's identity: archetype classification, ignition vector, focus domains, depth, completion state, momentum indicator, finalization beacon, and invoker. This is the permanent "fingerprint" of the conversation.

2

FAILURE SEMANTICS & INTEGRITY FLAGS

Zero-tolerance section. Must be fully populated ‚Äî no vague nulls. Captures incoherence flags, compression loss warnings, inference boundary alerts, resolution impossibility markers, failure severity (Low/Medium/High), and the Compression Fidelity Score (0.00‚Äì1.00 float).

3

COSMIC CORE & EMERGENCE

The conceptual heart of the artifact. Maps ontological constructs (key concepts), paradigm nodes (pivotal shifts in the thread), and emergent universals (cross-thread insights that transcend the immediate conversation).

4

DECISIONS & GRAFTS

Locks in what was decided. Architecture commits, heuristic branches (decisions that could go either way), and epistemic locks (facts treated as non-negotiable within this thread). Prevents revisiting settled ground.

5

OPEN VECTORS & THRUST

Forward-facing section. Unresolved queries, priority vectors with urgency tiers (Immediate / High / Medium / Low), and risk surfaces ‚Äî specific, named failure modes, not vague concerns.

6

THREAD TOPOLOGY (Conditional)

Only included if explicit cross-thread links exist. Maps parent, child, and sibling threads using [[Wikilink]] formatting for direct Obsidian Knowledge Graph compatibility. Domains tagged as #hashtags.

7

EXECUTION ARTIFACTS (Archive Mode Only)

Required when mode: archive. Records all generated outputs, tool usage patterns, reusability index, and integration notes ‚Äî creating a full provenance chain for the session's deliverables.

8

INVOCATION LOCK + Resurrection Hook

A 2‚Äì4 sentence sealing paragraph declaring the artifact immutable. Closes with the Resurrection Hook ‚Äî a single bootstrapping sentence formatted as > INGESTION: You are resuming a thread with the following established constraints: [...] ‚Äî designed to cold-start any LLM into full thread context.

Sample YAML Frontmatter + Resurrection Hook

--- (YAML Frontmatter) mode: re_entry protocol: "Universal Seed Summary Invoker" protocol_version: "1.3" timestamp: "2026-02-26 00:22:00 PST" --- ### INVOCATION LOCK ... sealing paragraph ... **Resurrection_Hook**: > INGESTION: You are resuming a thread with the following established constraints: USS v1.3 + ingestion-pack enhancements are the active protocol spec; four model-specific wrapper prompts have been generated with identical output schemas; validator.py requires two regex patches; all files target the DeeTorch/universal-seed-summary GitHub repository; next actions: patch validator, commit wrapper files, generate uss-v1.4.json.

Usage Guide

How to Use USS

USS requires no installation. It runs inside any capable LLM session. There are two components: the protocol prompt (the instruction layer) and the mode parameter (which controls output type).

1

Copy the USS Protocol Prompt

Retrieve the latest USS system prompt from the DeeTorch/universal-seed-summary GitHub repository. Use the universal ingestion-optimized version or one of the four model-specific wrappers (Claude XML, GPT-4o Markdown, Grok terminal, Gemini procedural).

2

Choose Your Invocation Mode

Set mode in the YAML frontmatter to one of three values: checkpoint (mid-session snapshot), re_entry (resuming an archived thread), or archive (final session seal with full execution artifact logging).

3

Paste Into Your LLM Session

Drop the USS protocol prompt at the end of your active conversation ‚Äî or at the start of a new session alongside a prior USS artifact. The model executes the protocol silently and generates the structured Markdown output.

4

Run the Validator

Pipe the output through validator.py in the tools/ directory. The validator checks all required fields, verifies the Compression_Fidelity_Score float, and confirms the Resurrection_Hook is present and properly formatted.

5

Store & Sync the Artifact

Commit the validated .md file to your GitHub repository and/or drop it into your Obsidian vault. The [[Wikilink]] formatting in the THREAD TOPOLOGY section is natively compatible with Obsidian's Knowledge Graph view ‚Äî cross-thread links render automatically.

6

Resurrect Any Thread, Anywhere

To resume a thread in any new session or any model, paste the Resurrection_Hook sentence as your opening message. The model is instantly rehydrated into the established constraints, decisions, and open vectors of the prior session ‚Äî zero context rebuild required.

The Three Invocation Modes

üì∏

Checkpoint

Mid-session snapshot. Captures current state without sealing the thread. Use when a conversation is active but you need a restore point before exploring a risky branch.

üîÅ

Re-Entry

Resumes a previously archived thread. The protocol ingests a prior USS artifact and continues from its established constraints. The most common daily-use mode.

üóÑÔ∏è

Archive

Final seal. Triggers Section 7 (Execution Artifacts). Produces a full provenance record of all generated outputs, tool usage, and integration notes. Use at true project completion.

Value Proposition

Why You Should Use USS

Every LLM session is ephemeral by default. USS converts that ephemerality into a structured, durable, cross-model knowledge layer.

üß†

Zero Context Rebuilding

The Resurrection Hook delivers the full established context of any session in a single sentence. No more "can you remind me where we left off?" prompts.

üîè

Audit-Safe Artifacts

Every USS output is a locked, immutable record. The Compression Fidelity Score quantifies nuance preservation. Failure flags surface exactly what was uncertain or compressed.

üåê

Cross-Model Portability

A USS artifact generated by Claude can be ingested by GPT-4o, Grok, or Gemini with identical fidelity. The schema is model-agnostic; only the wrapper scaffolding differs.

üóÇÔ∏è

Knowledge Graph Sync

Thread Topology uses [[Wikilink]] formatting natively. Drop USS artifacts into Obsidian and your entire project history becomes a navigable knowledge graph instantly.

‚öñÔ∏è

Truth-Over-Helpfulness Enforcement

USS prohibits the LLM from inventing, inferring, or hallucinating. Every field is thread-derived. If data is absent, it must be explicitly declared absent ‚Äî no silent omissions.

üõ†Ô∏è

Toolchain Integration

Ships with validator.py, indexer.py, and converter.py. Automate validation on commit, index artifacts for semantic search, and convert to any format your workflow requires.

The Core Problem USS Solves

LLM context windows are finite. Long threads degrade. Resuming sessions produces hallucinated continuations. Without a protocol, every new session is a clean slate ‚Äî and every insight built across hours of collaborative work can be silently lost. USS makes conversation memory a first-class engineered artifact, not an afterthought.

Model Obedience Profiles

The Four Model Wrappers

Identical USS output schema ‚Äî different delivery scaffolding. Each wrapper is optimized for the behavioral tendencies of its target model. Wrapper files live in templates/invocation-prompts/.

Cl
Claude
XML Wrapper ‚Äî uss-v1.4-claude.txt

Claude responds most reliably to XML-tagged instruction blocks. The USS Claude wrapper wraps the protocol in <uss_protocol_v1.4> tags, separating role, constraints, schema, and validation gate into discrete XML elements. Claude's strong instruction-following ensures near-zero schema deviation.

G+
ChatGPT / GPT-4o
Markdown Hierarchy ‚Äî uss-v1.4-chatgpt.txt

GPT-4o interprets Markdown heading hierarchies as instruction priority tiers. The USS GPT wrapper uses clean ## and ### heading structures with bold field labels. This approach capitalizes on GPT's strong Markdown-to-structure mapping and produces highly reliable YAML frontmatter compliance.

Gk
Grok (xAI)
Terminal Framing ‚Äî uss-v1.4-grok.txt

Grok responds well to terse, directive, terminal-style commands. The USS Grok wrapper frames the protocol as an executable script ‚Äî short imperative statements, // EXECUTE PROTOCOL NOW. as the closer. Risk note: this wrapper has not been live-tested at scale; verbose meta-commentary from Grok is a known risk surface.

Gm
Gemini
Procedural Steps ‚Äî uss-v1.4-gemini.txt

Gemini follows numbered procedural step formats most reliably. The USS Gemini wrapper presents each protocol phase as an explicit numbered step. A STEP 0: Do not narrate these steps. Execute them silently. directive is recommended to prevent Gemini from narrating the steps before execution ‚Äî a known behavioral risk.
