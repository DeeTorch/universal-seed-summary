# USS Engine v1.2 GitHub Issue Commands

These commands are safe to copy-paste into PowerShell when you are ready to create the issues. They do not run automatically from this document.

Prerequisites:

```powershell
gh auth status
gh repo set-default DeeTorch/universal-seed-summary
```

All commands use milestone `v1.2`. If the milestone does not exist yet, create it in GitHub first or remove the `--milestone "v1.2"` argument.

## Priority 1: Pre-Commit Secret Scanning

```powershell
$body = @'
## Problem

USS Engine can interact with live providers and private transcripts. The repository already ignores `.env`, `.venv/`, caches, and local output, but maintainers need stronger local guardrails before commits.

## Proposed Scope

- Add recommended `pre-commit` configuration.
- Include secret scanning with `detect-secrets` or an equivalent tool.
- Add checks for forbidden local output paths.
- Document setup and expected developer workflow.
- Keep scanning lightweight and optional for contributors who only run tests.

## Out Of Scope

- Rewriting Git history.
- Committing actual baseline secrets.
- Blocking all commits in environments without pre-commit installed.
- Changing runtime redaction behavior.

## Acceptance Criteria

- Documentation explains how to install and run pre-commit checks.
- Config checks `.env`, local output, caches, and common key patterns.
- Tests or CI checks confirm forbidden paths are not tracked.
- No secrets or generated provider outputs are committed.

## Files Likely Affected

- `.pre-commit-config.yaml`
- `.gitignore`
- `docs/security_hardening.md`
- `docs/release_process.md`
- `.github/workflows/tests.yml`
- New repository hygiene tests or scripts

## Risk Level

Low

## Suggested Test Coverage

- CI or script check for forbidden tracked paths.
- Test/sample scan against sanitized fake-secret fixtures only.
- Documentation verification for local setup commands.
- Regression check that `.env.example` remains allowed while `.env` remains ignored.

## Dependency Notes

Independent. This should land before feature work because it reduces the risk of committing provider keys, local output, or private transcript material during v1.2 development.
'@
gh issue create --repo DeeTorch/universal-seed-summary --title "Add pre-commit secret scanning and local output deny rules" --body $body --label "security,release-hardening,developer-experience,v1.2" --milestone "v1.2"
```

## Priority 2: Provider Validation Matrix Automation

```powershell
$body = @'
## Problem

`docs/provider_validation_matrix.md` is currently maintained manually. This is adequate for release freeze, but it can drift from actual provider smoke/checkpoint results.

## Proposed Scope

- Define a machine-readable provider validation result format.
- Add tooling to update or generate the provider validation matrix.
- Support static provider without network access.
- Support live providers only when credentials or local services are configured.
- Avoid writing secrets or raw provider responses to docs.

## Out Of Scope

- Running live provider tests in public CI by default.
- Creating new provider clients.
- Changing provider request/response behavior.
- Publishing raw private transcripts or prompts.

## Acceptance Criteria

- Static provider validation can run locally and in CI.
- Live provider validation is opt-in.
- Generated matrix includes provider, model, smoke test, checkpoint run, score, unsupported claims, status, and notes.
- No secret values are printed or written.
- Manual notes remain possible or are clearly separated.

## Files Likely Affected

- `docs/provider_validation_matrix.md`
- `src/uss_engine/cli.py`
- `src/uss_engine/reports.py`
- New validation automation tool/module
- `tests/test_provider_config.py`
- `tests/test_e2e_static.py`

## Risk Level

Medium

## Suggested Test Coverage

- Static provider matrix generation test.
- Tests for missing credentials.
- Tests that generated files do not include API-key-like values.
- Tests for preserving manual notes when applicable.

## Dependency Notes

Depends on the v1.1.2 provider validation matrix format staying stable. Should land before or alongside additional live-provider validation work. Does not require new provider clients.
'@
gh issue create --repo DeeTorch/universal-seed-summary --title "Automate provider validation matrix updates from smoke and checkpoint results" --body $body --label "provider-validation,automation,docs,v1.2" --milestone "v1.2"
```

## Priority 3: Semantic Evidence Matching

```powershell
$body = @'
## Problem

Current evidence anchoring is deterministic and local-first, using explicit references and lexical overlap. This is stable, but it can miss claims that are semantically supported with different wording.

## Proposed Scope

- Add an optional semantic matching layer behind the existing evidence-map contract.
- Preserve deterministic lexical matching as the default baseline.
- Define confidence outputs that can coexist with current evidence statuses.
- Add fixtures that compare lexical-only and semantic-assisted matching.
- Document when semantic matching is enabled and how to interpret it.

## Out Of Scope

- Replacing deterministic evidence mapping.
- Replacing current validation or scoring contracts.
- Adding hosted provider calls for evidence judging.
- Changing provider generation behavior.

## Acceptance Criteria

- Existing evidence-map JSON shape remains backward compatible.
- Semantic matching can be disabled.
- Existing tests still pass.
- New tests demonstrate at least one semantically supported claim missed by lexical matching.
- Unsupported factual claims remain detectable.

## Files Likely Affected

- `src/uss_engine/evidence.py`
- `src/uss_engine/scoring.py`
- `schemas/evidence_map.schema.json`
- `tests/test_evidence.py`
- `docs/evidence_anchoring.md`
- `docs/codebase_overview.md`

## Risk Level

High

## Suggested Test Coverage

- Unit tests for semantic match scoring thresholds.
- Regression tests for v1.1.1 metadata/null/assessment classifications.
- Backward-compatibility tests for evidence-map serialization.
- Negative tests proving unsupported claims do not become false positives.

## Dependency Notes

Should start after pre-commit safety is in place. Must preserve v1.1.1 evidence-classifier calibration behavior and the existing deterministic evidence-map contract.
'@
gh issue create --repo DeeTorch/universal-seed-summary --title "Add semantic evidence matching for USS artifact claims" --body $body --label "evidence,scoring,enhancement,v1.2" --milestone "v1.2"
```

## Priority 4: Local Archive Store

```powershell
$body = @'
## Problem

`uss run` produces a six-file artifact bundle, but there is no first-class local archive store for organizing, retaining, and querying completed runs.

## Proposed Scope

- Define a local archive directory layout.
- Add metadata conventions for run IDs, timestamps, provider, mode, and source thread.
- Support importing existing `uss run` output bundles.
- Keep archive storage local and file-based for v1.2.
- Document backup and privacy expectations.

## Out Of Scope

- Remote storage.
- Database server dependency.
- Cloud sync.
- Raw private transcript upload.
- Changing the existing `uss run` output contract.

## Acceptance Criteria

- Existing `uss run` output remains valid.
- Archived runs can be listed by metadata.
- Archive entries preserve the six canonical artifacts.
- Archive writes avoid secrets and respect ignored local output rules.
- Documentation explains local-only trust boundaries.

## Files Likely Affected

- `src/uss_engine/run.py`
- `src/uss_engine/reports.py`
- New archive module under `src/uss_engine/`
- `schemas/generation_run_report.schema.json`
- `tests/`
- `docs/`

## Risk Level

Medium

## Suggested Test Coverage

- Unit tests for archive path layout.
- Tests importing an existing output bundle.
- Tests listing archived runs by metadata.
- Tests rejecting incomplete bundles.
- Tests ensuring generated archive fixtures contain no secrets.

## Dependency Notes

Should preserve the six-artifact `uss run` output contract. Artifact index/search should depend on this or explicitly support the same archive layout.
'@
gh issue create --repo DeeTorch/universal-seed-summary --title "Add a local archive store for USS run artifacts" --body $body --label "archive,storage,enhancement,v1.2" --milestone "v1.2"
```

## Priority 7: GitHub Issue/PR Thread Summarizer

```powershell
$body = @'
## Problem

Project discussions often happen in GitHub issues and pull requests. USS Engine needs a safe path to summarize those threads without bypassing the existing transcript, redaction, validation, and evidence pipeline.

## Proposed Scope

- Define a GitHub thread input shape that converts issues, PR descriptions, review comments, and discussion comments into `NormalizedThread`.
- Preserve author, timestamp, URL, and event metadata where useful.
- Route normalized GitHub threads through the existing `uss run` pipeline.
- Document privacy and repository-access assumptions.

## Out Of Scope

- Implementing a GitHub bot.
- Mutating GitHub issues or PRs.
- Summarizing private repositories without explicit local credentials.
- Changing USS artifact validation rules.

## Acceptance Criteria

- GitHub-like fixture data normalizes into valid `NormalizedThread`.
- Redaction still runs before prompt compilation.
- Generated summaries can anchor evidence to normalized message IDs.
- Documentation explains credential and privacy boundaries.

## Files Likely Affected

- `src/uss_engine/transcript.py`
- New GitHub import/normalization module
- `src/uss_engine/cli.py`
- `tests/test_transcript.py`
- New GitHub fixture tests
- `docs/`

## Risk Level

Medium

## Suggested Test Coverage

- Tests for issue-only threads.
- Tests for PR with comments and reviews.
- Tests for author/timestamp metadata preservation.
- Tests for redaction of secrets in GitHub comments.
- Evidence-anchor tests against normalized GitHub message IDs.

## Dependency Notes

Can proceed independently if implemented as a normalization path into `NormalizedThread`. Should not bypass redaction, validation, evidence mapping, or the existing `uss run` pipeline.
'@
gh issue create --repo DeeTorch/universal-seed-summary --title "Normalize GitHub issue and PR threads into USS Engine inputs" --body $body --label "github,transcript-normalization,integration,v1.2" --milestone "v1.2"
```
